## Architecture
- **Framework/**: `AZSGameMode` (server rules, no mission logic yet), `AZSGameState` (P2, 2026-07-20: world clock â€” `TimeOfDayHours`/`DayCount`/utilities-shutoff timer â€” plus sleep/time-skip readiness aggregation across `PlayerArray`, see `Survival/` below), `AZSPlayerState` (health/ammo/kills, unpopulated), `AZSPlayerController` (Enhanced Input via `IMC_ZS_Default`+`IMC_ZS_MouseLook`, wired via `ConstructorHelpers::FObjectFinder` not BP CDO).
- **Player/**: `AZSPlayerCharacter` â€” TP/TopDown dual perspective (P1; FP rig/Bodycam/Inspect removed pre-P1, in git history). Move/look/jump/crouch/sprint/aim/fire/reload with full server-RPC+OnRep replication, config-driven weapon equip. `TopDown` is the default camera (fixed pitch, fixed yaw â€” the Q/E rotation feature was built then removed same-session at dev request); `ToggleCameraPerspective`/`IA_ToggleView` (`V`) swaps to `ThirdPerson` as the "OverShoulder" fallback per Decision 1. Hybrid cursor facing (`UpdateCursorFacing`) overrides actor rotation toward the mouse cursor's ground-plane projection, gated to aiming/attacking/interacting only. `Player/Animation/UZSAnimInstanceBase` = native TP AnimBP parent; FP subclass deleted. Known crouch-pose bug tracked in `Docs/SessionHandoff.md`. P4: `Server_Fire` does a real hitscan (`ECC_Visibility`, `UZSWeaponConfig::FireDamage`/`FireRange` from `SocketMuzzle`) and `Server_MeleeAttack` (bound to `IA_Attack`) is a weapon-independent bare-fist melee swing (sphere-overlap nearest-non-player-Pawn-in-front, `ECC_Pawn`, `Unarmed*` tunables) â€” both apply damage via `ApplyPointDamage` into the same `TakeDamage` pipeline P3 built. **Placeholder pending P5** ("Loadout & Unified Combat," `Docs/Phases/P5_CombatCompletion.md`): `IA_Attack` is meant to become one button whose effect dispatches per equipped item (unarmed/melee/ranged) once P5's `PrimaryHand`/`SecondaryHand`/hotbar loadout system exists â€” today it's unconditionally bare-fist, and `IA_Fire` is still a separate action from it.
- **Interaction/** (P1): `UZSInteractableComponent` â€” attach to any actor to make it interactable; `AZSPlayerCharacter::TryInteract`/`Server_Interact` is the server-authoritative entry point, `UpdateNearestInteractable` does a per-tick sphere-overlap scan. No visual world-prompt widget yet (UMG follow-up) â€” `OnNearestInteractableChanged` is the C++ hook for one.
- **Survival/** (new, P2, 2026-07-20): `UZSNeedsComponent` (subobject on `AZSPlayerCharacter`) â€” replicated Hunger/Thirst/Fatigue/Stamina, rates/curves from `UZSNeedsConfig`. Hunger/Thirst/Fatigue decay on an in-game-hour clock read from `AZSGameState`; Stamina is real-time (drains sprinting, regens idle), gates `StartSprint`/force-stops an active sprint at zero. `GetPerformanceMultiplier()` is the one accessor P3/P5 will wire healing rate/aim accuracy/attack recovery to later â€” no health hookup yet (`UZSHealthComponent` doesn't exist until P3). `UZSItemConfig` is P2's minimal eat/drink data contract, consumed via `Server_ConsumeItem`; no real inventory yet (P6). Sleep/time-skip lives across `AZSPlayerCharacter::RequestSleep`/`CancelSleepReady` (per-player ready flag) and `AZSGameState::UpdateSleepRequestState` (all-players-ready aggregation + clock advance); `IsSafeToSleep()` is stubbed `true` pending P4's zombies. Moodle UI (UMG) not built yet â€” `OnHungerChanged`/etc. are the C++ hooks.
- **Weapons/**: `AZSWeapon` (replicated config/fire-mode/ammo), `UZSWeaponConfig` (~22-field data contract + `FireNoiseRadius` for P4's noise system), `AZSMagazine` (cosmetic), notify classes `AN_ZS_UnlockActions`/`ANS_ZS_BlockADS`/`ANS_ZS_LeftHandGrip` (generalizes to melee/bandaging/barricading).
- **Combat/** (new, P3, 2026-07-20): `UZSHealthComponent` (subobject on `AZSPlayerCharacter`) â€” replicated `CurrentHealth` + 4 `FZSBodyZoneWound` (Head/Torso/Arms/Legs), each with a wound type (Scratch/Laceration/Fracture/Bite), bleed/dirty/splinted/amputated flags. All damage flows through `AZSPlayerCharacter::TakeDamage` (zone inferred from hit bone name, wound type from `DamageEvent.DamageTypeClass` via the `UZSDamageType_*` marker classes in `ZSDamageTypes.h`) into `Server_ApplyDamage` â€” nothing else mutates health directly, per the replication convention below. Bite wounds roll a hidden infection chance; on success, `EZSInfectionStage` progresses Noneâ†’Incubatingâ†’Queasyâ†’Feverâ†’Criticalâ†’death on `AZSGameState`'s game-hour clock (same pattern as `UZSNeedsComponent`'s decay). `Server_AmputateZone` (Arms/Legs only) clears an active infection at the cost of a permanent harsher zone penalty â€” "simplest version first" per `GameDevPlan.md` P3: any zone, solo-capable, no tool-item gate. `GetMobilityMultiplier()`/`GetAttackSpeedMultiplier()`/`GetReloadSpeedMultiplier()` are wired into `AZSPlayerCharacter`'s move speed and weapon fire-rate timer. Death (`OnDeath`) triggers `AZSPlayerCharacter::HandleDeath` â†’ a delayed `Destroy()` + `AGameModeBase::RestartPlayer` respawn-as-new-character flow. Treatment actions (`Server_ApplyBandage`/`Server_Disinfect`/`Server_Splint`) and item consumption both route through `AZSPlayerCharacter::Server_UseItem`, dispatching on `UZSItemConfig::EZSItemUseType` (extended in P3 to cover Bandage/Disinfectant/Splint alongside P2's Consumable).
- **Zombies/** (new, P4, 2026-07-20): `AZombieCharacter` â€” deliberately does NOT reuse `UZSHealthComponent` (that component's wound/infection/treatment machinery is player-specific); just a flat replicated `CurrentHealth` seeded from `UZSZombieConfig::MaxHealth`, same `TakeDamage`-only-entry-point rule. `Server_MeleeAttack` applies `ZombieConfig->AttackDamageTypeClass` (falls back to `UZSDamageType_Bite`) via `UGameplayStatics::ApplyPointDamage`, self-validating range/cooldown. `AZombieAIController` wires `AIPerception` (sight + hearing, both `UAISenseConfig_*` radii read from `ZombieConfig` at `OnPossess`) and calls `RunBehaviorTree(ZombieConfig->BehaviorTree)`. Drives the dev-authored `BT_Zombie`/`BB_Zombie` (`/Game/ZS/Enemy/AI/`, ShooterGame-derived) â€” `HandleTargetPerceptionUpdated` writes the `TargetActor`/`LastKnownLocation` Blackboard keys and calls `AZombieCharacter::SetChasing`; a 0.2s `Tick` maintains `bIsInMeleeRange`; `TriggerMeleeAttack`/`StartInvestigationTimer`/`StartIdleDwell` are the three functions `BT_Zombie`'s custom tasks call directly (cast `OwnerController` to this class). `UZSZombieConfig` covers speed/health/senses/damage/mesh/anim-class/BT/investigation+idle-dwell durations per zombie type â€” zero C++ branches for a second type, per the multi-weapon-config rule. `UZSNoiseSystem::ReportNoise` (a thin wrapper over `UAISense_Hearing::ReportNoiseEvent`) is P4's "load-bearing" noise system â€” wired into weapon fire (`FireNoiseRadius` per weapon) and sprint start.
- **Replication convention (mandatory from P1)**: `UPROPERTY(ReplicatedUsing=OnRep_X)` + `Server_X()` gated by `HasAuthority()` + `GetLifetimeReplicatedProps()`/`DOREPLIFETIME()` + `OnRep_X()` broadcasts delegate. Never poll replicated state directly.
- **Multi-weapon rule**: every weapon system must generalize to N weapon types via new `UZSWeaponConfig` instances, never new C++ branches.

## Workflow Efficiency (lessons, keep this section short)
- **Don't auto-attempt `Build.bat` while the editor's likely open.** It fails on the Live Coding lock nearly every time (see below) â€” a wasted round-trip for zero information. State that a rebuild is needed and let the dev trigger it (Rider/Ctrl+Alt+F11), rather than spending a call finding out again.
- **Check for a running editor with PowerShell's `Get-Process`, not Bash's `tasklist //FI`** â€” confirmed 2026-07-21 that a `tasklist` image-name filter run through the Bash tool silently matched nothing (wrong quoting through the POSIX-shell layer) while `UnrealEditor.exe` was, in fact, running; `Get-Process | Where-Object ProcessName -match "Unreal|LiveCoding"` caught it correctly. Also: **invoke `Build.bat` via the PowerShell tool, not Bash** â€” Bash's quoting mangles the batch file's internal `%~dp0`-style calls on this machine (`'C:\Program' is not recognized...`) even when the outer command looks correctly quoted; PowerShell runs it cleanly.
- **`Docs/SessionHandoff.md` is the sole owner of verification status** ("compiled?", "PIE-tested?", "what's next"). `CLAUDE.md`'s Architecture section and the `Docs/Phases/P<N>_*.md` files describe what exists / checklist state â€” don't restate compile/test status in more than one place, or every status change becomes a 3-file edit.
- **Use targeted reads for large, mostly-static docs** (`GameDevPlan.md`, long class headers) â€” `Grep` with context or `Read` with offset/limit â€” instead of reading the whole file when only one section is relevant.
- **Keep TaskCreate/TaskUpdate granularity coarse** â€” one task per major deliverable/system, not per file. Fine-grained tasks multiply tool-call round-trips without adding real visibility for a solo-dev session.

## MCP / Editor Tooling (lessons, keep this section short)
- `.mcp.json` only proposes the `unreal-mcp` server; actual enablement lives in `.claude/settings.local.json` (gitignored) â€” won't exist on a new machine until recreated.
- `describe_toolset` on a large toolset (e.g. `BlueprintTools`) can return 70K+ characters â€” don't call it broadly. Prefer calling a tool blind with a guessed schema first; the error response echoes the exact required schema, which is cheaper than a full describe dump.
- Simulated PIE input (keypresses, viewport clicks) via MCP does **not** reliably reach the pawn â€” confirmed 2026-07-20 (a simulated `W` never moved `CharMoveComp.Velocity`). Don't spend calls trying to force real gameplay-input verification headlessly; that needs the dev's hands, same as every past multiplayer/input check in this project.
- Poking a replicated/movement-authoritative property directly via `ObjectTools.set_properties` (e.g. `ACharacter::bIsCrouched`) can silently revert next tick if the owning component re-derives it every frame (e.g. from `bWantsToCrouch`) â€” a "successful" write isn't proof the state actually changed.
