## Tasks
- [x] `UZSHealthComponent`: 4 zones (Head, Torso, Arms, Legs), each a `FZSBodyZoneWound` (`ZSHealthTypes.h`). Wound types Scratch/Laceration/Fracture/Bite, ranked by severity (upgrade-only-if-more-severe on a fresh hit). Leg wounds degrade `GetMobilityMultiplier()` (wired into `AZSPlayerCharacter`'s `MaxWalkSpeed`); Arm wounds degrade `GetAttackSpeedMultiplier()`/`GetReloadSpeedMultiplier()` (fire-rate timer wired, reload not yet consumed anywhere). Bleed-over-time per bleeding zone, worse if dirty. All damage through `AZSPlayerCharacter::TakeDamage` → `Server_ApplyDamage` — zone inferred from hit bone name, wound type from `DamageEvent.DamageTypeClass` via `UZSDamageType_*` marker classes (`ZSDamageTypes.h`).
- [x] Treatment actions: `Server_ApplyBandage`(cleanliness flag)/`Server_Disinfect`/`Server_Splint` on `UZSHealthComponent`, dispatched via `AZSPlayerCharacter::Server_UseItem` reading `UZSItemConfig::EZSItemUseType`. **Not montage/notify-gated** — these are direct server mutators (same pattern as P2's `Server_ConsumeItem`), not wired into the `bIsBusy`/notify-timing action-choreography system reload uses. Hooking real medical montages into that system is deferred content/integration work (no montages exist yet).
- [x] Delayed-onset infection: `EZSInfectionStage` None→Incubating→Queasy→Fever→Critical→death, `HealthConfig->BiteInfectionChance` hidden roll on a Bite hit, stage durations in game-hours off `AZSGameState`'s clock. UI-ambiguity is a UI-layer responsibility (no UI built yet) - the enum itself isn't obscured in C++.
- [x] Emergency amputation: `AZSPlayerCharacter::AmputateZone` → `HealthComponent->Server_AmputateZone` (Arms/Legs only). Built the simplest version per this file's own instruction: any zone context, solo-capable, no tool-item requirement, valid any time before death - not gated behind a specific item/tool asset. Clears infection if amputating the infection-source zone; always applies the permanent `AmputatedZoneMultiplier` penalty regardless (amputating a merely-wounded, uninfected limb is a valid if wasteful choice, not blocked).
- [x] Player death → spectate/respawn-as-new-character flow: `UZSHealthComponent::Die` → `OnDeath` → `AZSPlayerCharacter::HandleDeath` (disables movement/collision/input on every machine) → server-only delayed `Destroy()` + `AGameModeBase::RestartPlayer` (spawns a fresh character, fresh Needs/Health state - not a heal-and-continue revive). No dedicated spectator camera/UI during the delay - `RespawnDelaySeconds` (default 5s) is just a timer, not a real spectate mode.

## Exit criteria
A scripted damage source can wound (with the correct gameplay-effect mapping), infect, and kill a player who mismanages treatment — and a player who amputates in time survives a bite that would otherwise have killed them. Second client sees everything correctly. See `Docs/SessionHandoff.md` for current status against this bar.
