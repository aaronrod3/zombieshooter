## Tasks

### Loadout system
**Design questions resolved 2026-07-21** (`GameDevPlan.md` §7 P5); **hotbar built 2026-07-21** (same day, `AZSPlayerCharacter`'s new "P5 - Loadout: real-time hotbar" section):
- [~] Two hand slots: `CurrentWeapon` **is** `PrimaryHand` (no rename — see note below), and it's the one `IA_Attack`/`HandleAttack` dispatches on, now filled/emptied in real time by the hotbar (see below). **`SecondaryHand` is deliberately still not built** — GameDevPlan.md §7 resolved it as independently-usable, but *how its own action gets triggered* is still an open follow-on question, and a slot nothing can trigger would just be dead code. Revisit once that question has an answer.
- [x] Empty `PrimaryHand` = bare-fist unarmed, always available, **and now the true default** — `BeginPlay` no longer auto-equips (`StartingWeaponConfig` is retired; `StartingHotbarLoadout`, an array, replaces it and seeds the hotbar instead). `RefreshBodyMeshFromWeapon` also now restores a cached `UnarmedBodyMesh` when unequipping, so the body mesh reverts visually too, not just the weapon actor.
- [x] Real-time hotbar: a fixed 9-slot `HotbarSlots` array (`TArray<UZSWeaponConfig*>`, replicated), seeded from `StartingHotbarLoadout` at `BeginPlay` — the same "author a reference directly on the character/BP" placeholder `StartingWeaponConfig` used, since P6's real inventory doesn't exist yet (see the unchecked item below). **Both input schemes built**: `SelectHotbarSlot`/`HandleHotbarSelect` (number keys 1-9, direct) and `CycleHotbar`/`HandleHotbarCycle` (scroll, skips empty slots, wraps). **Inputs not yet created**: `IA_HotbarSelect` (Axis1D, needs per-digit-key Scalar modifiers 1-9 in `IMC_ZS_Default`) and `IA_HotbarCycle` (Axis1D, mouse wheel) — the C++ finders degrade gracefully (same pattern as every other `IA_*`) until the dev creates them.
- [x] Equip/holster/switch takes real time: `Server_SelectHotbarSlot` → `SetBusy(true)` → a timer for the target weapon's `EquipTimeSeconds` (or the character's flat `UnequipTimeSeconds` when going bare-fist) → `CompleteHotbarSwitch` actually calls `EquipWeapon`/clears `CurrentWeapon` → `SetBusy(false)`. **v1 simplification, documented as such in the header**: one combined delay covers both holster-old and equip-new, not two separate notify-driven phases — no equip montage content exists yet to justify more than that. Re-pressing the already-equipped slot's number toggles unequip.
- [ ] Hotbar references P6's `UZSInventoryComponent` — still not true, still blocked on P6 existing. `StartingHotbarLoadout` is the same kind of placeholder `StartingWeaponConfig` always was, just an array now.

### Unified attack input
- [x] **Generalized `UZSWeaponConfig` with `EZSAttackType { Ranged, Melee }`** (`ZSWeaponTypes.h`) — no `Unarmed` value, since bare-fist isn't a weapon config instance (see below). Defaults to `Ranged` so every config authored before this didn't need updating.
- [x] **`AZSPlayerCharacter::HandleAttack` dispatches on `CurrentWeapon`'s config**: `Ranged` → routes into the existing `HandleFireStarted`/auto-fire-timer machinery unchanged; `Melee` → `Server_WeaponMeleeAttack` (real per-weapon stats, see below); no weapon → `Server_MeleeAttack`, the flat bare-fist `Unarmed*` tunables. `HandleAttackStopped` (bound to `IA_Attack` Completed) stops the auto-fire timer if one was started - a no-op after a melee swing.
- [x] **`IA_Fire` retired as a separate input binding** - `SetupPlayerInputComponent` no longer binds it; `IA_Attack` alone drives both paths now. This is what the dispatch above needed to actually fix the double-trigger.
- [x] **Real per-weapon melee stats on `UZSWeaponConfig`** (2026-07-21, same night as the hotbar): `MeleeDamage`/`MeleeRange`/`MeleeAttackInterval`/`MeleeDamageTypeClass`/`MeleeMontage`, one-for-one mirrors of `AZSPlayerCharacter`'s `Unarmed*` fields. The overlap-scan/hit-application logic itself was extracted into a shared `PerformMeleeSwing` helper (bare-fist and weapon melee were duplicating ~40 identical lines) - `Server_MeleeAttack` (bare-fist) and the new `Server_WeaponMeleeAttack` (real weapon) both call it with their own stats. **Still no melee `UZSWeaponConfig` instance actually authored** - the fields exist and the dispatch is wired, but nothing equips a `Melee`-typed weapon to test against yet (needs a `DA_ZS_WeaponConfig_<MeleeItem>` created in-editor, which needs a mesh/attachment approach decided first - see the note below).
- [x] **Weapon durability-lite built** (2026-07-21): `UZSWeaponConfig::MaxDurabilityHits` (0 = unbreakable, the default - every gun and any untuned melee config), `AZSWeapon::CurrentDurability`/`Server_ConsumeDurabilityHit()`. **Chosen v1 interpretation, not previously specified**: breaking a weapon doesn't just unequip it, it also nulls that weapon out of its own `HotbarSlots` entry - durability lives on the `AZSWeapon` actor instance (there's no per-item-instance state in the hotbar itself, just a `UZSWeaponConfig*` reference), so leaving the slot filled would let re-selecting it spawn a fresh, full-durability weapon from the same config. This is a real limitation worth revisiting once P6's inventory gives items actual per-instance state.
- [x] **Simple hit-reaction knockback built** (2026-07-21): `ApplyHitKnockback` (shared static helper, `LaunchCharacter` on any hit `ACharacter`) wired into `PerformMeleeSwing` (both melee paths, `MeleeKnockbackStrength`/`UnarmedMeleeKnockbackStrength`) and `Server_Fire` (`FireKnockbackStrength`). **This is physical-only, not a real stagger/interrupt AI state** - a zombie gets shoved but its Behavior Tree doesn't know or care; a full "knockdown" (interrupts the BT, a recovery window) would need `BT_Zombie` editing, which is explicitly out of scope for an unsupervised session (Blueprint graph edits are exactly the Live Coding corruption risk class logged in `CLAUDE.md`). Revisit once someone can supervise a PIE test of a BT change.
- [ ] Melee weapon types through this pipeline: swing timing via the existing notify system (stats exist, montage-notify integration untested - no montage authored); **shove + stomp still not built** - shove needs its own input action decision (no key assigned - not guessed at unsupervised) and stomp needs a "downed/staggered zombie" state that doesn't exist anywhere in the AI yet (see the knockback note above); stamina economy tuned against P2's `UZSNeedsComponent` also not wired (melee doesn't cost stamina yet). **Variety resolved 2026-07-21**: curated 4–6 archetypes (one per feel-category — blunt/edged/improvised/heavy, not exact final names), not PZ's full breadth. Exact items TBD from `Content/LowPolyWeapons/`/`Content/Mega_Survival_Tools/` (already sourced, see `GameDevPlan.md` §5) - **blocked on deciding how a melee weapon attaches/displays**, since `TP_Mesh` is a full-body skeletal mesh swap (the gun-holding pose model) and a one-handed bat/pipe likely wants a simpler held-prop attachment instead; not resolved this session, flagged for the dev.
- [ ] Firearms integration with noise + zombie mass: ammo scarcity tuning (needs real playtest data, not guessed numbers) still not done; hit-reaction/knockdown is **partially built** (physical knockback only - see above).
- [x] Weapon durability-lite (melee breaks; no repair sim v1) — **built**, see above (was already decided as a design target, not re-litigated 2026-07-21 - now actually implemented).

## Pre-P5 groundwork already built (2026-07-21, to unblock P3/P4 testing before this phase started)
`AZSPlayerCharacter::Server_MeleeAttack` (bare-fist sphere-overlap melee) and `Server_Fire`'s hitscan both exist and work. Originally bound to separate inputs (`IA_Attack` → melee, `IA_Fire` → gunfire) with no dispatch between them - that's what the "Unified attack input" tasks above fixed the same day, once PIE testing surfaced the double-trigger. `IA_Attack` was named ahead of this phase specifically so the dev's placeholder input action wouldn't need a second rename once real dispatch landed. See `Docs/Phases/P4_Zombies.md` and `Docs/SessionHandoff.md` for what's already PIE-testable.

## Exit criteria
A player starts unarmed, equips a weapon from the hotbar in real time, and one Attack button does the right thing whether they're bare-handed, swinging a bat, or holding a rifle. On top of that, the PZ death loop exists — greed + noise + stamina mismanagement kills a player who had every tool to survive.
