### Session-ending discovery — Enhanced Input mappings likely never actually took effect (READ THIS FIRST NEXT SESSION)
Dev reported `IA_ToggleView` (`V` key) not working in PIE, and that the `IMC_ZS_Default` editor panel only shows the original 3 actions (Jump/Move/Look), not the 10 new Phase 2 ones from this session's M4 work — **even after closing/reopening the asset tab and a full editor restart.**

Investigating via MCP found the actual cause, and it's bigger than a UI refresh glitch: **`UInputMappingContext` has two separate mapping arrays**, both discovered via `ObjectTools.get_properties`:
- **`mappings`** (top-level property) — this is what every `set_properties` call this session (Phase 1 *and* Phase 2 M4) wrote to and read back successfully. Currently holds all 30 correct entries for `IMC_ZS_Default` (verified again at end of session) and both correct entries for `IMC_ZS_MouseLook`.
- **`defaultKeyMappings.mappings`** (nested under a `defaultKeyMappings` struct) — a **completely separate, independently-populated array that this session never wrote to at all.** Reading it back at end of session shows it already contained its own full set of bindings *before any of this session's work*: Jump (SpaceBar/`Gamepad_FaceButton_Bottom`), Move (W/S/A/D **+ arrow keys** + `Gamepad_Left2D`), Look (`Gamepad_Right2D`) on `IMC_ZS_Default`; MouseLook on `Mouse2D` (a single composite key, not the split MouseX/MouseY this session used) on `IMC_ZS_MouseLook`.

**This means the editor UI (and quite possibly the actual runtime Enhanced Input subsystem) may be reading `defaultKeyMappings`, not `mappings`.** If that's true: Phase 1's WASD/jump/mouse-look "worked" not because of anything built this session, but because `defaultKeyMappings` already had a complete, functional stock binding set from whenever the template/asset was first created — and **every Enhanced Input change made across this entire session (Phase 1 verification *and* all of Phase 2 M4's 30 mappings) may have landed in a property nothing actually consults.**

**Not yet confirmed, not yet fixed — this is exactly where next session should start:**
1. Check Epic's actual UE5.8 `InputMappingContext.h` source (or the Enhanced Input docs) to understand what `Mappings` vs. `DefaultKeyMappings` vs. `MappingProfileOverrides` actually mean and which one the runtime `UEnhancedInputLocalPlayerSubsystem` reads at `AddMappingContext` time. Don't guess again — this session already lost time to two unverified guesses (see `CLAUDE.md`'s MCP section pattern: verify, don't assume).
2. Once the authoritative property is identified, either move all 32 of this session's mappings onto it, or write to both if that's actually how the system is meant to work (e.g. `defaultKeyMappings` as a designer-set base and `mappings` as something else like a runtime cache — order of authority not yet established).
3. Re-verify every new Phase 2 input action (`Fire`/`Aim`/`Reload`/`Crouch`/`Sprint`/`ToggleView`/`FireModeSwitch`/`Inspect`/`MagCheck`/`SwitchGrip`) actually fires in PIE once fixed, not just `ToggleView`.
4. This is unrelated to (and doesn't invalidate) the separate `ConstructorHelpers` CDO-reconstruction caveat below, which is still real and still needs a restart to resolve independently.
