### RESOLVED (session 4, later) — AZSWeapon ammo/magazine/fire-mode setup was dead code (BeginPlay-vs-InitializeFromConfig ordering bug)
Verifying the now-actually-spawning weapon in PIE, found `CurrentMagazineAmmo`/`CurrentReserveAmmo` were `0`/`0` despite `DA_ZS_WeaponConfig_AssaultRifle` correctly having `MagazineCapacity=30`/`StartingReserveAmmo=90` (ruled out a data gap before assuming one). Root cause in [ZSWeapon.cpp](Source/ZombieShooter/Weapons/ZSWeapon.cpp): `AZSWeapon::BeginPlay()` contained ALL of the config-dependent setup (ammo seeding, fire-mode selection, laser-attachment spawn, default grip, **and the magazine spawn + reserve-hidden visibility M7 needed to verify**), gated behind `if (!CurrentConfig) return;`. But `AZSPlayerCharacter::EquipWeapon()` calls `SpawnActor<AZSWeapon>()` *then* `InitializeFromConfig(Config)` afterward — and `SpawnActor` fires the new actor's `BeginPlay()` synchronously, immediately, whenever the world has already begun play (true for every real case: the starting weapon equips from the player's own `BeginPlay()`, after the world has begun play). So `BeginPlay()` always ran with `CurrentConfig` still null, hit the early return, and none of that block ever executed — not just for the starting weapon, for *any* future mid-game weapon swap too. The header's own comment on `InitializeFromConfig` even said "Must be called before BeginPlay," documenting the intended order without noticing `SpawnActor` doesn't honor it.

**Fix:** moved the entire config-dependent block from `BeginPlay()` into `InitializeFromConfig()` (the function actually guaranteed to run, since `EquipWeapon` calls it explicitly right after spawn), and removed the now-empty `BeginPlay()` override from both `.h`/`.cpp`. **Compiled and PIE-tested clean by the dev** — `CurrentMagazineAmmo`/`CurrentReserveAmmo` now read `30`/`90` on spawn, matching `DA_ZS_WeaponConfig_AssaultRifle` exactly.

### MCP tooling notes learned this session (for future sessions using unreal-mcp against a live PIE session)
- **`ObjectTools.set_properties` on a Blueprint's CDO does not propagate to newly-spawned PIE instances until `BlueprintTools.compile_blueprint` is called explicitly.** Hit this directly: set `StartingWeaponConfig` on a CDO, restarted PIE, the freshly-spawned actor still read the old (unset) value. Recompiling the Blueprint fixed it. Always compile after a raw CDO property edit, before trusting a PIE test.
- **`EditorAppToolset.CaptureViewport` renders from the *editor's own stored level-viewport camera* (a virtual scene capture), not a screen-grab of what's on-screen during PIE.** Its `captureTransform` param must be passed as literal JSON `null` to mean "use the current camera" — an explicit `{}` is interpreted as an actual identity transform (world origin), producing a nonsensical shot from inside level geometry at (0,0,0). Even with `null`, it uses the editor's free-cam position (wherever it was last left, e.g. a stock top-down level-overview position), not the player's in-game camera — there's no built-in "look through this component" option, so reproducing an exact player POV requires manually reading the camera component's world transform and passing it explicitly (attempted this; still didn't yield a usable shot — the render layer also appears to use some non-Lit debug view mode with editor-only camera-frustum gizmos baked in). For genuine "what does the player see" verification, this tool is not yet a reliable substitute for the dev's own eyes in PIE.
- `CaptureViewport`/`describe_toolset` calls can return over a million characters of base64 image data or schema text that blows the tool's own token limit — comes back as a saved file path instead. Decode/inspect these via PowerShell (`ConvertFrom-Json` + `[Convert]::FromBase64String`), not Bash (no `python3` available in this environment's Git Bash).
- Renaming a Blueprint via `AssetTools.move` correctly fixes up references stored in other assets/CDOs *and* in already-loaded level actor instances (confirmed: `WorldSettings.DefaultGameMode` auto-updated). It can still leave a small redirector-stub file at the old path — `AssetTools.delete` on that stub path may not fully remove it from disk even though it reports success; fall back to `git rm` for a leftover stub once you've confirmed (via readback) nothing still points at it.

### RESOLVED (session 4, 2026-07-13) — Enhanced Input `mappings` vs `defaultKeyMappings.mappings`
Session 3 ended having found two separate mapping arrays on `UInputMappingContext` without knowing which was authoritative. Confirmed this session by reading the actual UE5.8 engine source (`Engine/Plugins/EnhancedInput/Source/EnhancedInput/{Public/InputMappingContext.h,Private/InputMappingContext.cpp,Private/EnhancedInputSubsystemInterface.cpp}`) — no more guessing:

- **The top-level `Mappings` property is `UE_DEPRECATED(5.7, "Use the DefaultKeyMappings struct instead.")`.** It still exists and is still a `UPROPERTY(config, ...)` (so MCP's `set_properties`/`get_properties` can read/write it without error), but nothing in the Enhanced Input runtime reads it.
- **`DefaultKeyMappings.Mappings` (the nested struct property) is authoritative.** `UInputMappingContext::GetMappings()`, `GetMappingsForProfile()`, `ForEachKeyMapping()`, `MapKey()`/`UnmapKey()`/`UnmapAll()` all read/write `DefaultKeyMappings.Mappings` exclusively. Confirmed the actual runtime call chain: `IEnhancedInputSubsystemInterface::RebuildControlMappings()` (`EnhancedInputSubsystemInterface.cpp:988`, the function that runs when `AddMappingContext` is called) calls `MappingContext->GetMappingsForProfile(...)`, which returns `DefaultKeyMappings.Mappings`.
- **Why the deprecated field silently held valid-looking data with zero effect:** `UInputMappingContext::PostLoad()` has a one-time migration (`DefaultKeyMappings.Mappings = MoveTemp(Mappings)`) but it's gated on `GetLinkerCustomVersion(...) < FFortniteSeasonBranchObjectVersion::EnhancedInputMappingContextProfileMappingsUpdate`. Both `IMC_ZS_Default`/`IMC_ZS_MouseLook` were created fresh in UE5.8, so they already serialize at/above that version — the migration branch never fires, no matter what's written into the deprecated `Mappings` array. It's a dead write target, not a fallback or a cache.

**Fix applied and verified via MCP (not just a bare success return):**
1. Read both properties live off `IMC_ZS_Default`/`IMC_ZS_MouseLook` via `ObjectTools.get_properties` before touching anything. Confirmed session 3's memory was accurate: the deprecated `mappings` held all 30 correct entries (Jump/Move/Look + the 10 Phase 2 M4 actions, keyboard+gamepad) on `IMC_ZS_Default` and both correct split-axis entries on `IMC_ZS_MouseLook`; `defaultKeyMappings.mappings` held stale pre-existing template data (Jump/Move/Look only, Move including arrow keys, gamepad as deadzone-modified 2D composite axes; MouseLook as a single `Mouse2D` composite) that nothing this project built.
2. Wrote the full correct 30-entry set (`IMC_ZS_Default`) and 2-entry set (`IMC_ZS_MouseLook`) onto `defaultKeyMappings.mappings` via `ObjectTools.set_properties` (had to clear the array to `[]` first in a separate call — a single call that both resized and repopulated the array errored with "ArrayAdd: elements changed alongside the size change; insertion points are ambiguous").
3. Re-read both properties back — byte-for-byte match with what was written.
4. Cleared the now-dead deprecated `mappings` array to `[]` on both assets (pure cleanup, avoids a future session finding two populated arrays again) and called `AssetTools.save_assets` on both. `is_dirty` confirmed `false` afterward.

**Known, deliberate side effect of the fix — flagging for the dev, not treated as a blocker:** the fix replaced (not merged with) the old `defaultKeyMappings.mappings` content, since the instruction was to move this session's actual designed bindings onto the authoritative property. That means **arrow-key support for `IA_Move` and the deadzone-modified gamepad 2D composite axes (`Gamepad_Left2D`/`Gamepad_Right2D`) are gone**, replaced by the split-axis gamepad bindings (`Gamepad_LeftX`/`Gamepad_LeftY` etc., no deadzone modifier) that were already designed and written last session. Keyboard WASD/mouse-look/all 10 Phase 2 actions are unaffected. If gamepad deadzone or arrow-key movement matters, that's a follow-up, not a regression from working behavior — the split-axis mappings were never live before this fix.

### RESOLVED (session 4, later still) — AnimGraph MCP-DSL risk spike run, decisive negative result on the DSL round-trip
Tested against Infima's own real reference AnimBP (`ABP_TFA_FP_BaseCharacter` — read-only inspection, never wrote to it, per `CLAUDE.md`'s off-limits rule on `Content/InfimaGames/`):
- **`BlueprintTools.read_graph_dsl` returns an empty string for AnimGraph content** — tried three different graph refs (the root `AnimGraph`, a nested state machine `SM_LocomotionStanding`, and an individual state's own graph `AnimStateNode_1.Running`), all empty.
- **Ruled out a general tool malfunction first:** `read_graph_dsl` on the same Blueprint's `EventGraph` produced rich, correct output (full event/function/cast/branch structure). The DSL text format is built entirely around exec/statement semantics (`event`/`fn`/`bind`/`if`/`switch` — confirmed via `get_graph_dsl_docs`, which never mentions poses, states, or blend nodes at all) — it has no representation for AnimGraph's pose-dataflow paradigm (`FPoseLink`/`FComponentSpacePoseLink`), so it isn't a partial gap, it's a category mismatch.
- **But the lower-level node-discovery API is NOT blind to AnimGraphs** — `find_node_types` against the AnimGraph (empty filter, ~1.7M chars of results) surfaced deep, specific awareness: `Animation|StateMachines|StateMachine` (the actual "add state machine" node), per-state introspection nodes generated for this exact graph's real state names (`StateWeight(Stand)`, `StateWeight(Running)`, `StateWeight(Sprinting)`, etc.), `LayeredBoneBlend|GetNumPoses`/`ConverttoLayeredBoneBlend`, a full `StateMachine|*`/`BlendSpacePlayer|*` function library, and — notably — our own `UZSAnimInstanceBase`/`ZSFirstPersonAnimInstance`'s existing `BlueprintPure` getters (`GetTargetGripAlpha`, `GetGripPoseBlendSpeed`, `GetInputMoveVector`, etc.) already discoverable through the same mechanism.
- **Conclusion for M8:** the convenient DSL text round-trip (`read_graph_dsl`/`write_graph_dsl`) cannot be used to build or edit AnimGraph pose-flow content (state machines, blend nodes, pose links) — don't attempt it. The lower-level structured node API (`create_node`/`connect_pins`/`add_node_pin`/`set_node_position`, working with real pose-link pins instead of DSL statements) has genuine node-type awareness and may still be viable for building AnimGraph content node-by-node, but that's a substantially bigger, unproven lift than a text round-trip and hasn't been attempted — actually constructing a state machine this way is M8's own first real test, not something to pre-verify further here. EventGraph-side AnimBP logic (variable updates from the owning character, custom events, casts — same shape as `EventBlueprintUpdateAnimation`) remains fully buildable via the DSL without issue.
- **Practical read for M8 planning:** budget for `create_node`/`connect_pins` node-by-node AnimGraph construction being slow and failure-prone, and don't be surprised if some of it (particularly wiring up state machines/transitions) ends up needing the dev's own hands in the AnimGraph editor — `CLAUDE.md`'s existing caveat on M8 ("may need the dev's own hands if MCP can't reach AnimGraph editing") was the right instinct.

Asked to confirm this was correct and check for missing steps — it was **not** correct, confirmed two independent ways:

1. **Infima's own guide is explicit.** `10_Custom_Character_And_Weapon_Import.md`'s Guide A: assign any custom/duplicated mesh onto **`SKEL_TFA_Mannequin`** (their own bundled skeleton) — "No IK Retargeter step is needed here... this workflow assumes direct skeleton-sharing compatibility... not cross-skeleton retargeting." `01_Installation_And_Project_Structure.md` confirms `Common/Characters/Mannequins/` "contains the shared mannequin skeleton every included animation is assigned to" — i.e. one skeleton, used by everything in the pack. `SK_Mannequin` (Epic's own stock UE5 Manny skeleton, unrelated to Infima) is never mentioned as a valid target anywhere in the guide.
2. **Verified directly via MCP, not just the docs.** `SkeletalMeshTools.get_skeleton` on Infima's own `SKM_Manny_Simple` (this project's already-working TP/FP body mesh) *and* `SKM_FP_Manny_Simple` (the FP-arms source mesh) both resolve to `SKEL_TFA_Mannequin` — every piece of Infima content in this project, including everything already working, has always used `SKEL_TFA_Mannequin`. `SKM_FirstPerson_Manny_Simple` (the new duplicate) resolves to `SK_Mannequin` instead — an outlier.

**Bone hierarchies are identical** (`get_bone_names` on a mesh bound to each skeleton: 145 bones, same names, same order, byte-for-byte diffed — confirmed via a real diff, not eyeballing; spot-checked `get_bone_parent` on the FABRIK-required bones `ik_hand_r`/`hand_r` too, same parent chain both sides). Infima has simply bundled their own exact copy of Epic's stock UE5 Manny rig under their own asset name — a common marketplace-pack practice. But being bone-identical does **not** make two separate `USkeleton` assets interchangeable in UE5 by default; that needs the engine's explicit "Compatible Skeletons" feature configured (not set up here, and not what Infima's guide uses/recommends at all).

**Concrete, functional proof this actually breaks something (not just a documentation nicety):** `SkeletalMeshTools.get_socket_names` shows `SOCKET_CameraFP` (the exact socket `AZSPlayerCharacter::EnableFirstPersonPerspective` attaches `FirstPersonCamera` to, per this session's earlier M7 fix), plus `SOCKET_HelmetCamera`/`SOCKET_ChestCamera`/`SOCKET_Hand_L_Throw`/`headSocket`, are **skeleton-level sockets defined on `SKEL_TFA_Mannequin` itself**, not per-mesh — both `SKM_Manny_Simple` and `SKM_FP_Manny_Simple` (two otherwise-unrelated meshes) show the identical extra-socket set, proving they inherit from the shared skeleton rather than each carrying their own copies. The reassigned duplicate lost every one of them, and picked up `SK_Mannequin`'s own unrelated sockets (`HandGrip_R`/`HandGrip_L`) instead. Left as-is, `FirstPersonCamera` (and the Bodycam perspective's socket lookups) would silently fail to attach — `AttachWeaponToActiveMesh`'s and the camera-attach code's missing-socket guards no-op rather than crash, so this would likely show up as "camera looks wrong" with no error, not an obvious failure.

**Verdict — duplicating the FP arms mesh at all is the right call** (Infima's own Guide A step 3 recommends duplicating before reassigning, "keeps the original clean for rollback" — and using the *dedicated* FP arms mesh (`SKM_FP_Manny_Simple`) instead of the full body mesh is a genuine improvement: `DA_ZS_WeaponConfig_AssaultRifle.FP_Mesh` currently still points at the full-body `SKM_Manny_Simple`, confirmed live via MCP — using the full body mesh in first person was itself an under-noticed M6 config gap, not the intended setup). **Only the skeleton target needs correcting: reassign `SKM_FirstPerson_Manny_Simple` to `SKEL_TFA_Mannequin`, not `SK_Mannequin`.**
