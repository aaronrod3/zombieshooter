While attempting to visually verify M7 in PIE (via `unreal-mcp`'s `EditorAppToolset.StartPIE`/`CaptureViewport`, discovered this session ŌĆö see the MCP tooling notes below), found that `Lvl_ThirdPerson`'s `WorldSettings.DefaultGameMode` was `BP_ThirdPersonGameMode`, not any `BP_ZS_*` class. Blueprint CDO overrides always beat a C++ constructor's default, so `AZSGameMode`'s `ConstructorHelpers::FClassFinder` pointing at `BP_ZS_PlayerCharacter` ([ZSGameMode.cpp:21](Source/ZombieShooter/Framework/ZSGameMode.cpp:21), from M6) was silently shadowed and never took effect.

- **Fix (dev-approved consolidation, not a unilateral call ŌĆö asked first via AskUserQuestion):** set `StartingWeaponConfig = DA_ZS_WeaponConfig_AssaultRifle` on `BP_ThirdPersonCharacter`'s CDO, deleted the unused `BP_ZS_PlayerCharacter` decoy, then renamed+relocated all three real Blueprints to match convention: `BP_ThirdPersonGameMode` ŌåÆ `Content/ZS/Framework/BP_ZS_GameMode`, `BP_ThirdPersonCharacter` ŌåÆ `Content/ZS/Characters/BP_ZS_PlayerCharacter`, `BP_ThirdPersonPlayerController` ŌåÆ `Content/ZS/Framework/BP_ZS_PlayerController`. ### RESOLVED (session 4, later) ŌĆö AZSWeapon ammo/magazine/fire-mode setup was dead code (BeginPlay-vs-InitializeFromConfig ordering bug)
Verifying the now-actually-spawning weapon in PIE, found `CurrentMagazineAmmo`/`CurrentReserveAmmo` were `0`/`0` despite `DA_ZS_WeaponConfig_AssaultRifle` correctly having `MagazineCapacity=30`/`StartingReserveAmmo=90` (ruled out a data gap before assuming one). Root cause in [ZSWeapon.cpp](Source/ZombieShooter/Weapons/ZSWeapon.cpp): `AZSWeapon::BeginPlay()` contained ALL of the config-dependent setup (ammo seeding, fire-mode selection, laser-attachment spawn, default grip, **and the magazine spawn + reserve-hidden visibility M7 needed to verify**), gated behind `if (!CurrentConfig) return;`. But `AZSPlayerCharacter::EquipWeapon()` calls `SpawnActor<AZSWeapon>()` *then* `InitializeFromConfig(Config)` afterward ŌĆö and `SpawnActor` fires the new actor's `BeginPlay()` synchronously, immediately, whenever the world has already begun play (true for every real case: the starting weapon equips from the player's own `BeginPlay()`, after the world has begun play). So `BeginPlay()` always ran with `CurrentConfig` still null, hit the early return, and none of that block ever executed ŌĆö not just for the starting weapon, for *any* future mid-game weapon swap too. The header's own comment on `InitializeFromConfig` even said "Must be called before BeginPlay," documenting the intended order without noticing `SpawnActor` doesn't honor it.

**Fix:** moved the entire config-dependent block from `BeginPlay()` into `InitializeFromConfig()` (the function actually guaranteed to run, since `EquipWeapon` calls it explicitly right after spawn), and removed the now-empty `BeginPlay()` override from both `.h`/`.cpp`. **Compiled and PIE-tested clean by the dev** ŌĆö `CurrentMagazineAmmo`/`CurrentReserveAmmo` now read `30`/`90` on spawn, matching `DA_ZS_WeaponConfig_AssaultRifle` exactly.

### MCP tooling notes learned this session (for future sessions using unreal-mcp against a live PIE session)
- **`ObjectTools.set_properties` on a Blueprint's CDO does not propagate to newly-spawned PIE instances until `BlueprintTools.compile_blueprint` is called explicitly.** Hit this directly: set `StartingWeaponConfig` on a CDO, restarted PIE, the freshly-spawned actor still read the old (unset) value. Recompiling the Blueprint fixed it. Always compile after a raw CDO property edit, before trusting a PIE test.
- **`EditorAppToolset.CaptureViewport` renders from the *editor's own stored level-viewport camera* (a virtual scene capture), not a screen-grab of what's on-screen during PIE.** Its `captureTransform` param must be passed as literal JSON `null` to mean "use the current camera" ŌĆö an explicit `{}` is interpreted as an actual identity transform (world origin), producing a nonsensical shot from inside level geometry at (0,0,0). Even with `null`, it uses the editor's free-cam position (wherever it was last left, e.g. a stock top-down level-overview position), not the player's in-game camera ŌĆö there's no built-in "look through this component" option, so reproducing an exact player POV requires manually reading the camera component's world transform and passing it explicitly (attempted this; still didn't yield a usable shot ŌĆö the render layer also appears to use some non-Lit debug view mode with editor-only camera-frustum gizmos baked in). For genuine "what does the player see" verification, this tool is not yet a reliable substitute for the dev's own eyes in PIE.
- `CaptureViewport`/`describe_toolset` calls can return over a million characters of base64 image data or schema text that blows the tool's own token limit ŌĆö comes back as a saved file path instead. Decode/inspect these via PowerShell (`ConvertFrom-Json` + `[Convert]::FromBase64String`), not Bash (no `python3` available in this environment's Git Bash).
- Renaming a Blueprint via `AssetTools.move` correctly fixes up references stored in other assets/CDOs *and* in already-loaded level actor instances (confirmed: `WorldSettings.DefaultGameMode` auto-updated). It can still leave a small redirector-stub file at the old path ŌĆö `AssetTools.delete` on that stub path may not fully remove it from disk even though it reports success; fall back to `git rm` for a leftover stub once you've confirmed (via readback) nothing still points at it.

### RESOLVED (session 4, 2026-07-13) ŌĆö Enhanced Input `mappings` vs `defaultKeyMappings.mappings`
Session 3 ended having found two separate mapping arrays on `UInputMappingContext` without knowing which was authoritative. Confirmed this session by reading the actual UE5.8 engine source (`Engine/Plugins/EnhancedInput/Source/EnhancedInput/{Public/InputMappingContext.h,Private/InputMappingContext.cpp,Private/EnhancedInputSubsystemInterface.cpp}`) ŌĆö no more guessing:

- **The top-level `Mappings` property is `UE_DEPRECATED(5.7, "Use the DefaultKeyMappings struct instead.")`.** It still exists and is still a `UPROPERTY(config, ...)` (so MCP's `set_properties`/`get_properties` can read/write it without error), but nothing in the Enhanced Input runtime reads it.
- **`DefaultKeyMappings.Mappings` (the nested struct property) is authoritative.** `UInputMappingContext::GetMappings()`, `GetMappingsForProfile()`, `ForEachKeyMapping()`, `MapKey()`/`UnmapKey()`/`UnmapAll()` all read/write `DefaultKeyMappings.Mappings` exclusively. Confirmed the actual runtime call chain: `IEnhancedInputSubsystemInterface::RebuildControlMappings()` (`EnhancedInputSubsystemInterface.cpp:988`, the function that runs when `AddMappingContext` is called) calls `MappingContext->GetMappingsForProfile(...)`, which returns `DefaultKeyMappings.Mappings`.
- **Why the deprecated field silently held valid-looking data with zero effect:** `UInputMappingContext::PostLoad()` has a one-time migration (`DefaultKeyMappings.Mappings = MoveTemp(Mappings)`) but it's gated on `GetLinkerCustomVersion(...) < FFortniteSeasonBranchObjectVersion::EnhancedInputMappingContextProfileMappingsUpdate`. Both `IMC_ZS_Default`/`IMC_ZS_MouseLook` were created fresh in UE5.8, so they already serialize at/above that version ŌĆö the migration branch never fires, no matter what's written into the deprecated `Mappings` array. It's a dead write target, not a fallback or a cache.

**Fix applied and verified via MCP (not just a bare success return):**
1. Read both properties live off `IMC_ZS_Default`/`IMC_ZS_MouseLook` via `ObjectTools.get_properties` before touching anything. Confirmed session 3's memory was accurate: the deprecated `mappings` held all 30 correct entries (Jump/Move/Look + the 10 Phase 2 M4 actions, keyboard+gamepad) on `IMC_ZS_Default` and both correct split-axis entries on `IMC_ZS_MouseLook`; `defaultKeyMappings.mappings` held stale pre-existing template data (Jump/Move/Look only, Move including arrow keys, gamepad as deadzone-modified 2D composite axes; MouseLook as a single `Mouse2D` composite) that nothing this project built.
2. Wrote the full correct 30-entry set (`IMC_ZS_Default`) and 2-entry set (`IMC_ZS_MouseLook`) onto `defaultKeyMappings.mappings` via `ObjectTools.set_properties` (had to clear the array to `[]` first in a separate call ŌĆö a single call that both resized and repopulated the array errored with "ArrayAdd: elements changed alongside the size change; insertion points are ambiguous").
3. Re-read both properties back ŌĆö byte-for-byte match with what was written.
4. Cleared the now-dead deprecated `mappings` array to `[]` on both assets (pure cleanup, avoids a future session finding two populated arrays again) and called `AssetTools.save_assets` on both. `is_dirty` confirmed `false` afterward.

**Known, deliberate side effect of the fix ŌĆö flagging for the dev, not treated as a blocker:** the fix replaced (not merged with) the old `defaultKeyMappings.mappings` content, since the instruction was to move this session's actual designed bindings onto the authoritative property. That means **arrow-key support for `IA_Move` and the deadzone-modified gamepad 2D composite axes (`Gamepad_Left2D`/`Gamepad_Right2D`) are gone**, replaced by the split-axis gamepad bindings (`Gamepad_LeftX`/`Gamepad_LeftY` etc., no deadzone modifier) that were already designed and written last session. Keyboard WASD/mouse-look/all 10 Phase 2 actions are unaffected. If gamepad deadzone or arrow-key movement matters, that's a follow-up, not a regression from working behavior ŌĆö the split-axis mappings were never live before this fix.
